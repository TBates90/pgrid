Pentagon–Hex Corner Wedge Artefact in pgrid: Updated Findings and Resolution Plan
What the latest renders show

Across your three new globe renders (detail_rings = 1, 3, 5), the primary remaining defect is no longer a broad “shear/pinch” across the pentagon-centred neighbourhood. Instead, it presents as a small triangular “wedge” artefact located at or very near a tile vertex, i.e. where multiple Goldberg tiles meet (often visually adjacent to a pentagon–hex junction).

Two characteristics matter for diagnosis:

    The artefact is highly localised (a small triangle), not a smooth global distortion over an entire tile.
    The artefact becomes less visually dominant as detail_rings increases (the wedge remains possible, but the sub-face structure gets smaller so the wedge reads as a tiny blemish rather than a big distortion).

That pattern is consistent with a corner/vertex sampling or per-vertex UV inconsistency problem (often introduced in the “atlas → mesh UV sampling” stage), rather than the earlier planar-stitch closure distortion that primarily affects a whole outer ring around a pentagon.
What has changed in main branch since the initial fix set

Your main branch now contains the key mitigations from the earlier plan, and they’re the right mitigations for the original “pent-centred neighbourhood pinch” class of defect.

    Pentagon-centred composites skip neighbour↔neighbour closure. The intent is: for 5-neighbour rings you stop trying to “force planar closure” among the outer neighbours, while still keeping the centre↔neighbour seams correct (the only seams the tile truly must satisfy). This change (and its rationale) is captured in the project’s own diagnosis notes.
    Hex composites now have an overlap-quality guard before snapping/stitching any outer neighbour pair, which prevents “closest midpoint” matches from being forcibly collapsed when the edges don’t actually overlap well.
    Atlas seam enforcement post-pass exists for polygon-cut atlases. The current polygon-cut atlas builder includes an atlas-space seam stitch that averages boundary pixels along shared Goldberg edges, reducing residual edge discontinuities caused by independent resampling.

These address the dominant geometric distortion mode at pentagon neighbourhoods (the one you originally described), and they also add a safety net for residual edge mismatches.
Most likely remaining root cause

Given the shape of the remaining artefact (a small triangular wedge at/near a multi-tile vertex) and the fact it diminishes in salience as detail_rings increases, the most likely insertion point is not the composite stitcher any more. It is the 3D mesh UV sampling path, specifically the interaction of:

    UV “inset + clamp” behaviour applied during mesh subdivision, and
    vertex deduplication-by-position inside each subdivided tile fan.

Why this points at the mesh stage

The v2 renderer subdivides each tile’s triangle fan and (optionally) clamps UVs to an inset polygon to avoid sampling outside the textured region. The code path is in subdivide_tile_mesh() where UVs are generated by barycentric interpolation and then clamped when uv_clamp_polygon is provided.

Critically, subdivide_tile_mesh() then deduplicates vertices using only 3D position (rounded) to merge shared vertices.

That combination is safe only if the UV computed for a shared position is identical in all sectors that produce that position.

But when you clamp to an inset polygon:

    Two computations that land on the same 3D position can be pushed to different nearest points on the inset polygon (especially near corners where two edges compete).
    Dedup then keeps one of those UVs (whichever is encountered first) and reuses it for triangles that expected the other.
    The result is exactly what you’re seeing: a tiny triangular patch (often one or a few micro-triangles) whose UVs “jump” to the wrong edge/corner sample, and therefore its colour/texture looks like a wedge intrusion.

Why you’re especially seeing it near pentagon junctions

Pentagon neighbourhoods are exactly where:

    tile vertex valence differs (5 around a pent corner region),
    UV polygon shapes are more irregular in local arrangements,
    and small differences in “which edge is nearest” near a corner can become more sensitive.

Separately, your rendering toolchain likely uses scripts/render_globe_from_tiles.py, which in v2 mode calls render_globe_v2() without exposing UV inset as a user-tunable parameter; so you are implicitly using the renderer defaults.
Fast experiments to confirm the insertion point

These are quick, high-signal checks that distinguish “atlas content” issues from “mesh sampling” issues.
Toggle UV inset/clamp during globe rendering

Add a CLI parameter to scripts/render_globe_from_tiles.py to pass through uv_inset_px into render_globe_v2(). Right now that script always uses the renderer’s default inset behaviour.

Run the same atlas with:

    uv_inset_px = 0.0
    uv_inset_px = 0.5
    uv_inset_px = 1.5 (current default)

Expected outcome if this hypothesis is correct:
The wedge artefact should drop dramatically (often disappear) as you approach 0.0. If it does, the defect is confirmed to be mesh UV clamping + dedup interaction, not the stitch/warp pipeline itself.
Compare “warped tile” outputs to on-globe wedge locations

You already have tooling intended for pinpointing where the artefact enters the pipeline. The repo documentation explicitly recommends comparing “stitched composite” vs “warped tile” stages for pent tiles.

If warped tile PNGs look clean at the corners, but the globe shows wedges, that again confirms the issue is post-atlas (mesh stage).
Comprehensive resolution plan

This plan is structured to (a) confirm the true insertion point with minimal effort, (b) fix the root cause in the most direct place, and (c) add “belt and braces” protections against corner-only artefacts that seam-stitching along edges won’t fully eliminate.
Make UV inset controllable and lower-risk for polygon-cut atlases

    Expose uv_inset_px in render_globe_from_tiles.py.
    Add a new CLI arg (defaulting to current behaviour to avoid breaking users), then pass it into the v2 renderer call.

    Minimal patch sketch (illustrative):

    python

    # in argparse:
    parser.add_argument("--uv-inset-px", type=float, default=1.5,
                        help="Inset UV polygon (px) before optional clamping")

    # in render_globe_v2 call:
    render_globe_v2(
        payload, atlas_path, uv_layout,
        title=title,
        subdivisions=args.subdivisions,
        width=args.width,
        height=args.height,
        uv_inset_px=args.uv_inset_px,
    )

    Adopt a safer default for polygon-cut atlases (recommended once verified).
    If testing shows uv_inset_px=0 removes the wedge without reintroducing bleed, consider defaulting to 0 for polygon-cut pipelines specifically, because polygon-cut already provides:
        per-tile gutters in the atlas, and
        seam averaging in atlas space.

    Practically: keep the renderer default at 1.5 globally, but set the script default to 0.0 when you’re in “polygon-cut atlas” mode.

Make UV clamping compatible with vertex deduplication

Even if you keep some amount of inset (useful defensively), you can eliminate the “shared position → conflicting UV → wedge triangle” failure mode by ensuring that boundary/near-boundary UVs remain consistent.

    Do not clamp UVs on the polygon boundary.
    In subdivide_tile_mesh(), detect points on the polygon boundary edge of each fan triangle and skip clamping for those points.

    In your triangle fan, points on the polygon boundary edge are exactly those with barycentric weight b0 == 0 relative to the centre vertex. The code already explicitly computes b0, b1, b2 in the subdivision loop, so this is easy and cheap.

    Conceptually:
        If b0 == 0: it lies on the tile boundary edge; do not clamp.
        If b0 > 0: it’s interior; clamping is allowed (but often unnecessary).

    This keeps the boundary UVs coherent and dramatically reduces the chance that two fan sectors compute different “nearest edge” clamp solutions for the same spatial point.

    If clamping remains, adjust vertex dedup to be attribute-safe.
    The current per-tile dedup uses position only. If UVs can differ at the same position (because clamping is non-linear), you have two robust options:
        Option A (preferred): make clamping deterministic so that shared positions always map to identical UVs (boundary skip is the simplest form of this).
        Option B (failsafe): include UV (or a quantised UV) in the dedup key, so vertices that coincide in XYZ but not in UV are not merged. This removes the possibility of “wrong UV chosen for merged vertex”, at the cost of a larger vertex buffer.

Add a corner-aware post-pass if needed

Your seam enforcement currently operates along shared edges. That is necessary, but it can still leave corner-only artefacts because a vertex is a multi-way junction (especially around pentagon regions).

    Implement vertex-junction blending in atlas space (optional but powerful).
    Add a second post-pass after _stitch_atlas_seams() that:
        groups all tiles incident to a shared globe vertex,
        samples a small neighbourhood (e.g. radius 2–3 px) around that vertex inside each tile’s atlas slot,
        writes the average back to all incident tiles.

    This is the atlas-space analogue of the existing edge blending and acts as a strong “last mile” fix for corner-only wedges. It’s also easy to make conditional (only run if a corner discontinuity metric exceeds a threshold).

    The existing edge stitch infrastructure in tile_uv_align.py already has:
        access to uv_layout,
        access to authoritative per-tile UV polygons,
        and an adjacency finder used for edges.

Keep the earlier pentagon-specific geometry work as a separate track

Do not re-introduce planar closure for pentagon-centred composites. That was the correct move for the original distortion class.

However, the repo’s own issue tracker notes a separate, known pentagon characteristic: the “zigzag bias” in the pentagon macro boundary can cause uneven UV cut behaviour. There is even an unused helper (_smooth_pentagon_corners) discussed as an optional strategy, but it carries a seam-risk tradeoff at pentagon vertices.

This should be treated as a second, optional quality upgrade after the wedge artefact (corner micro-triangle) is fixed, because that artefact is currently the more visually disruptive and more likely to be caused by UV clamping/dedup.
Validation criteria

To avoid “fix one view, regress another”, validate at three levels:

    Atlas correctness: warped tiles should remain fully covered (no sentinel and no cval holes), and edge seam stitch should still produce consistent edges.
    Globe render correctness: with uv_inset_px=0 (or boundary-safe clamping), the triangular wedge artefact should disappear at the problematic pentagon-adjacent junctions in colour-debug mode, across detail_rings = 1, 3, 5.
    Regression protection: keep hex↔hex continuity intact (your already-good case), which is exactly what the current composite behaviour is designed to preserve.
